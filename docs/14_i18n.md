# Phase i18n - Internationalisation 8 langues

Spec d'execution detaillee pour ajouter 8 langues au projet Shinobi no Sho avec strict 100% de couverture textuelle, dans un budget Sonnet 4.6 Batch API de $25.

Ce document est la source de verite pour la phase i18n. Tout developpement, manuel ou automatique, doit s'y conformer.

## Vision en une phrase

Le joueur choisit sa langue au premier lancement, ce choix est sauve durablement, et 100% du texte affiche par le jeu (UI, narration, descriptions canon, missions, items, dialogues, biographie, rumors, OpenAPI doc) est dans cette langue, sauf les termes explicitement preserves en romaji (jutsu, noms canon, kekkei genkai, etc.).

## Langues supportees

| Code | Nom natif | Source canonical |
|------|-----------|------------------|
| `en` | English | **Langue source de verite** (toutes les autres en derivent) |
| `fr` | Francais | Existante (donnees scrapees Narutopedia FR) |
| `es` | Espanol | Generee via Sonnet 4.6 |
| `ja` | 日本語 | Generee via Sonnet 4.6 |
| `zh` | 中文 (mandarin simplifie) | Generee via Sonnet 4.6 |
| `ko` | 한국어 | Generee via Sonnet 4.6 |
| `pt-BR` | Portugues brasileiro | Generee via Sonnet 4.6 |
| `de` | Deutsch | Generee via Sonnet 4.6 |

Pourquoi EN comme source de verite et non FR alors que les donnees sont en FR ? Parce que les chaines UI les plus naturelles sont anglaises (cles courtes, conventions standard) et parce que nous re-generons toutes les autres langues a partir d'EN une fois les chaines UI extraites. Le contenu canon (wiki_sections, motivations, etc.) reste source FR mais sera traduit FR -> {EN + 6 autres} via Sonnet.

## Principes fondateurs i18n

### Strict 100% de couverture

Chaque chaine visible par le joueur DOIT etre dans la langue choisie. Aucun texte FR ne doit fuiter dans une UI EN/ES/JA/ZH/KO/PT/DE. Cela inclut :

- Tout panneau Rich CLI (titres, bordures, contenus)
- Tous les prompts (`Prompt.ask(...)`)
- Toutes les commandes slash (descriptions dans `/help`)
- Tous les messages d'erreur visibles joueur
- Tous les outcomes d'action (`"Reussite eclatante"` -> `"Brilliant success"` etc.)
- Toutes les biography events (`"Promotion : genin -> chunin"`)
- Toutes les descriptions de missions, items, summons, item effects
- Toutes les rumors generees par tick_scheduler
- Toutes les narrations LLM (Qwen3-4B re-prompte dans la langue cible)
- Toutes les descriptions API (`summary`, `description` des routes FastAPI)
- Tous les wiki_sections affiches (background, personality, abilities, etc.)
- Tous les datasets Phase H (motivations, factions, divergence_points, narrative_patterns, timeline_enriched)

### Termes preserves volontairement (allowlist)

Reste en **romaji** dans toutes les langues :

- Noms de jutsu (`Katon: Goukakyuu no Jutsu`, `Rasengan`, `Chidori`)
- Noms de personnages canon (`Uchiha Itachi`, `Senju Hashirama`)
- Noms de villages (`Konohagakure`, `Sunagakure`)
- Noms de clans (`Uchiha`, `Hyuuga`, `Senju`)
- Noms d'organisations (`Akatsuki`, `Anbu`)
- Termes techniques (`chakra`, `jinchuuriki`, `bijuu`, `kekkei genkai`, `kekkei mora`, `senjutsu`, `fuinjutsu`, `genjutsu`, `taijutsu`, `ninjutsu`)
- Rangs (`genin`, `chunin`, `jonin`, `kage`, `anbu`)
- Natures (`katon`, `suiton`, `fuuton`, `doton`, `raiton`, `mokuton`, `hyouton`, `youton`, etc.)
- Les noms de bijuu (`Kyuubi`, `Hachibi`, etc.) et leurs forms

Cette liste de termes intraductibles est centralisee dans `data/i18n/glossary.json` et le narrateur LLM la recoit en context-injection systematique.

### EN comme langue par defaut

Au premier lancement, EN est selectionne par defaut dans le picker. Tu peux changer juste apres ou plus tard via le menu.

## Architecture technique

### Stockage des preferences

Cross-platform via `platformdirs` (deja utilisable, pas de nouvelle dependance lourde).

```python
# Linux: ~/.config/shinobi-no-sho/preferences.json
# macOS: ~/Library/Application Support/shinobi-no-sho/preferences.json
# Windows: C:/Users/<USER>/AppData/Local/shinobi-no-sho/preferences.json
```

Schema :

```json
{
  "language": "en",
  "first_launch_completed": true,
  "language_chosen_at": "2026-05-08T09:30:00Z",
  "schema_version": 1
}
```

Si le fichier n'existe pas ou contient un `language` invalide, le picker s'affiche au prochain lancement et set `first_launch_completed: true`.

### Module `shinobi.i18n`

Nouveau package : `src/shinobi/i18n/`

```
src/shinobi/i18n/
  __init__.py             # Exporte t(), set_language(), get_language()
  loader.py               # Charge les .json par langue + fallback EN
  preferences.py          # Read/write preferences.json via platformdirs
  glossary.py             # Liste des termes preserves (romaji)
  catalog.py              # Mappings cles -> traductions (singleton runtime)
```

API publique :

```python
from shinobi.i18n import t, set_language, get_language

# Lookup avec interpolation
t("cli.menu.welcome")
# -> "Welcome to Shinobi no Sho" (en)
# -> "Bienvenue dans Shinobi no Sho" (fr)
# -> "シノビの書へようこそ" (ja)

t("engine.outcome.partial_success_with_target", target="Itachi")
# -> "Partial success against Itachi"

set_language("ja")  # change runtime

get_language()  # -> "ja"
```

Cle plate : `"<scope>.<area>.<key>"`. Convention :

- `cli.*` : couche presentation Rich/Typer
- `engine.*` : moteur (outcomes, biography, etc.)
- `canon.*` : labels canon (rangs, natures, categories)
- `mission.*`, `item.*`, `summon.*` : catalogues
- `validator.*`, `error.*` : messages d'erreur joueur
- `narrator.system_prompt` : prompts LLM
- `api.<route>.summary` : descriptions OpenAPI

### Format des fichiers de traduction

Un fichier JSON par langue dans `data/i18n/<lang>.json` :

```json
{
  "cli.menu.welcome": "Welcome to Shinobi no Sho",
  "cli.menu.quit": "Quit",
  "cli.menu.new": "New character",
  "engine.outcome.full_success": "Brilliant success",
  "engine.outcome.partial_success": "Partial success",
  "engine.outcome.minor_failure": "Minor failure",
  "engine.outcome.catastrophic_failure": "Catastrophic failure",
  "engine.outcome.contextual_impossibility": "Impossible in this context",
  ...
}
```

Fallback : si une cle est absente d'un fichier `<lang>.json`, on prend la valeur de `en.json`. Si elle est absente d'EN aussi (regression), on retourne la cle elle-meme (`cli.menu.unknown` -> `"cli.menu.unknown"`) avec un log warning.

### Wiki sections / contenu long

Les wiki_sections etant trop massifs pour les fichiers UI, ils sont stockes separement :

```
data/i18n/wiki/
  en/
    uzumaki_naruto.json   # { "background": "...", "personality": "...", ... }
    uchiha_itachi.json
    ...
  fr/
    uzumaki_naruto.json
    ...
  ja/
  zh/
  ko/
  pt-BR/
  de/
```

Strategie hybride :

- **Top-100 personnages les plus joues** : pre-traduits via Sonnet Batch. Ils sont determines par presence dans `deep_motivations` + `political_forces.factions[].leader_id` + `divergence_points.involved_canon_ids` + heuristique notoriete (kekkei_genkai owners).
- **Reste (~1260 chars)** : traduits a la volee via Qwen3-4B local au premier acces, cache en `data/i18n/wiki/<lang>/<canon_id>.json` ainsi que dans `<save>/wiki_translation_cache/`.

### Phase H datasets

5 datasets actuellement en FR (genere via Sonnet pour le projet initial) :

- `deep_motivations.json` (50 chars cles)
- `political_forces.json` (49 factions)
- `divergence_points.json` (21 events critiques)
- `narrative_patterns.json` (14 patterns canon)
- `timeline_events_enriched.json` (294 events)

Dossier final :

```
data/canon/i18n/
  en/deep_motivations.json
  en/political_forces.json
  ...
  fr/...                   # deja existant
  ja/...
  zh/...
  ko/...
  pt-BR/...
  de/...
```

Le loader `shinobi.canon.loader._load_phase_h_datasets` lit le dossier `<lang>/` selon la langue active.

### LLM prompts

6 prompts systeme par langue :

```
data/i18n/prompts/
  en/narrator.txt
  en/pathfinder.txt
  en/character_interpreter.txt
  en/world_resolver.txt
  en/director.txt
  en/tension_analyst.txt
  fr/narrator.txt
  ...
```

Le client Qwen recoit le bon prompt selon `get_language()`. Le glossary (termes preserves) est injecte automatiquement en footer du prompt pour que le LLM ne traduise PAS les jutsu / personnages.

Le contexte RAG reste en FR (le retriever Chroma indexe le corpus FR existant). Quand on injecte un fact dans le prompt LLM en mode EN/JA/etc., le LLM doit traduire mentalement le contexte FR vers la sortie. **Quand la qualite est mauvaise, on bascule vers traduction batch des facts via Sonnet** (Phase i18n.7 partielle, $0 supplementaires si on rentre dans le buffer).

### Saves et i18n

Une save persiste les chaines verbatim au moment ou elles sont produites. Si le joueur change de langue en cours de partie :

- Les `biography_log` deja existants restent dans la langue d'origine
- Les nouveaux events sont dans la nouvelle langue
- Les `goals.description_player` (saisie joueur libre) restent verbatim
- Les `goals.interpretation_canonical` sont re-translates au prochain `/path` ou `/turn` (via Qwen)

Acceptable selon ta confirmation Q11.

### Texte joueur custom (objectifs, intentions)

Le joueur tape en langue libre. Pour eviter qu'un goal "Je veux devenir Hokage" tape en FR pollue un contexte LLM EN :

- Au moment du save : detection langue via Qwen (call rapide ~50 tokens)
- Si langue detectee != langue config : traduction Qwen vers langue config
- Cache dans `Goal.description_player_translated[lang]`
- L'original reste accessible via `description_player_original`

Le texte affiche au joueur est toujours dans sa langue config.

## Phases d'execution

Sequencees. Chaque phase a un objectif, des fichiers livrables, un critere de sortie, et une duree estimee.

### Phase i18n.1 : Infrastructure i18n core (1 jour)

Objectif : module `shinobi.i18n` fonctionnel avec lookup, fallback EN, hot-swap runtime.

Livrables :

- `src/shinobi/i18n/__init__.py` exposant `t`, `set_language`, `get_language`, `available_languages`
- `src/shinobi/i18n/loader.py` qui charge `data/i18n/<lang>.json` paresseusement
- `src/shinobi/i18n/preferences.py` qui read/write `preferences.json` via `platformdirs`
- `src/shinobi/i18n/glossary.py` avec la liste des ~50 termes preserves
- 8 fichiers stub vides : `data/i18n/{en,fr,es,ja,zh,ko,pt-BR,de}.json` avec juste `{"_schema": "i18n_v1"}`
- `tests/unit/test_i18n_core.py` : 15 tests (chargement, fallback, runtime switch, glossary)

Critere de sortie :

- `t("test.key", x=1)` fonctionne avec interpolation
- Switch `set_language("ja")` met a jour le catalogue actif sans relancer
- Cle absente -> retourne fallback EN puis cle elle-meme avec log warning

### Phase i18n.2 : Settings + first-launch flow (0.5 jour)

Objectif : picker au 1er lancement et menu pour changer.

Livrables :

- Modification `src/shinobi/cli/app.py` callback : check `preferences.first_launch_completed`, si False, lance le picker
- `src/shinobi/cli/language_picker.py` : Rich panel avec les 8 langues affichees dans leur propre langue
- Slash command `/language` ajoutee dans `src/shinobi/cli/play.py` `_handle_meta` qui re-affiche le picker
- Endpoint API `PUT /preferences/language` (body : `{"language": "ja"}`)
- Endpoint API `GET /preferences` (lecture)
- `tests/unit/test_language_picker.py` : 8 tests
- `tests/unit/test_api_preferences.py` : 6 tests

Critere de sortie :

- Premier `shinobi` (ou `shinobi.exe` futur) sans `preferences.json` affiche le picker
- Une fois choisi, sauve durablement, ne re-demande plus
- `/language` reset persiste le nouveau choix
- API `PUT /preferences/language` updated le runtime + le fichier

### Phase i18n.3 : Extraction des chaines UI (1.5 jour)

Objectif : remplacer toutes les chaines en dur par des appels `t(...)`.

Modules a passer au peigne fin :

- `src/shinobi/cli/app.py` (~30 chaines)
- `src/shinobi/cli/menu.py` (~30 chaines)
- `src/shinobi/cli/character_creation.py` (~70 chaines, dont prompts interactifs)
- `src/shinobi/cli/canon_incarnation.py` (~25 chaines)
- `src/shinobi/cli/play.py` (~150 chaines, le gros morceau, slash commands inclus)
- `src/shinobi/cli/display.py` (~20 chaines)
- `src/shinobi/cli/streaming_display.py` (~10 chaines)

Total estime : **~335 cles**.

Methode :

1. Script `scripts/extract_i18n_strings.py` qui scanne avec AST tous les `console.print(...)`, `Panel(...)`, `Prompt.ask(...)`, `typer.echo(...)`, `typer.confirm(...)` avec un literal francais.
2. Genere `data/i18n/_extracted.json` avec proposition de cle (`<module>.<context>.<num>`)
3. Review humaine pour renommer les cles vers une nomenclature semantique (`cli.menu.welcome` au lieu de `cli.app.0`)
4. Le script remplace ensuite les literals par `t("<key>")` automatiquement
5. Tests run pour s'assurer que rien casse

Livrables :

- `scripts/extract_i18n_strings.py`
- `data/i18n/en.json` peuple avec ~335 cles UI en anglais (traduit depuis FR via Sonnet, voir Phase i18n.5)
- `data/i18n/fr.json` peuple avec les chaines originales FR
- Tous les modules CLI passes en `t(...)` calls
- Tests existants (1524) doivent toujours passer

Critere de sortie :

- `grep -rn "console.print.*\".*[éèàù]" src/shinobi/cli/` retourne 0 resultat (aucun literal FR restant)
- `pytest` 1524 tests pass (avec lang=en par defaut, les tests existants assertent contre les chaines EN)

### Phase i18n.5 : Translation batch UI/catalogs/prompts via Sonnet (0.5 jour)

Objectif : generer en/es/ja/zh/ko/pt-BR/de pour les ~500 cles UI + 6 prompts, **avec EN comme source pour les 7 cibles**.

Note : EN est genere depuis FR via Sonnet aussi (puisque les originaux sont FR), donc en realite on traduit FR -> {EN, ES, JA, ZH, KO, PT-BR, DE} = 7 langues cibles.

Methode :

1. Script `scripts/i18n_batch_translate.py` qui genere les requetes Anthropic Batch API
2. 1 batch par paire (langue cible × dataset). Datasets : `ui` (500 cles), `catalogs` (165 cles), `prompts` (6 textes longs)
3. Total : 7 langues × 3 datasets = 21 batches en parallele
4. Cout reel : ~$1.40 (UI + catalogs + prompts)
5. Output : `data/i18n/{en,es,ja,zh,ko,pt-BR,de}.json`
6. Validation auto : verification de presence de toutes les cles, presence du glossary intacte (regex sur les termes preserves)

Prompt Sonnet (extrait) :

```
You are a professional video game translator specializing in the Naruto universe.

Translate the following JSON keys from French to {target_language}.

CRITICAL RULES:
1. Output VALID JSON with the exact same keys, only values translated.
2. PRESERVE these terms in original romaji form: {GLOSSARY_LIST}
3. Keep formatting tokens intact: {variable_name}, [bold]...[/bold], etc.
4. Maintain register and tone consistent with a serious narrative video game.
5. Output ONLY the translated JSON, no preamble.

Source (French):
{json_content}
```

Livrables :

- `scripts/i18n_batch_translate.py`
- 7 fichiers `data/i18n/<lang>.json` complets
- 6 fichiers `data/i18n/prompts/<lang>/*.txt` × 7 langs = 42 fichiers
- Rapport batch : `research/i18n-batch-report.md` avec cout reel + couverture + warnings

Critere de sortie :

- Aucun fichier `<lang>.json` ne manque de cle (verification automatique)
- Glossary preserve dans 100% des entrees (regex sur tokens romaji)
- Cout reel <= $2

### Phase i18n.6 : Wiki sections strategy (1 jour code + 6h batch run)

Objectif : pre-traduire le top-100 chars + infrastructure on-the-fly Qwen pour le reste.

#### 6.A : Pre-translation top-100 via Sonnet

Selection des 100 chars (script `scripts/i18n_select_top100.py`) :

- Tous les chars dans `deep_motivations.keys()` (~50)
- Tous les `political_forces.factions[].leader_id` (~30, chevauchement attendu avec deep_motivations)
- Tous les `divergence_points.involved_canon_ids` (~40, chevauchement)
- Completer jusqu'a 100 par notoriete (`canon.characters` ayant `kekkei_genkai` non vide ou `tailed_beast` non null)

Pour chacun de ces 100 chars, traduction de **3 sections wiki essentielles** :

1. `background` (histoire perso)
2. `personality` (traits)
3. `abilities` (techniques et capacites)

Les autres sections (`appearance`, `trivia`, `quotes`, `part_i`, `part_ii`, `boruto`) ne sont pas pre-traduites (Qwen on-the-fly).

Script `scripts/i18n_translate_wikis.py` :

- 100 chars × 3 sections × 7 langs = 2100 batches
- Token estime moyen : 400 mots × 1.5 = 600 tokens FR par section
- Total : 100 × 3 × 600 = 180K tokens FR a traduire vers 7 langs
- Cout estime : $13.20 (voir budget)

Livrables :

- `data/i18n/wiki/<lang>/<canon_id>.json` × 100 chars × 7 langs = 700 fichiers
- Rapport : `research/i18n-wiki-translation-report.md`

#### 6.B : Qwen on-the-fly pour le reste

Script `src/shinobi/i18n/wiki_translator.py` :

- Quand `/canon/characters/{id}` est appele en mode != FR et que `data/i18n/wiki/<lang>/<id>.json` n'existe pas, lance traduction Qwen3-4B locale
- Cache resultat dans `data/i18n/wiki/<lang>/<id>.json` (persistant cross-saves)
- Latence 1er acces : 5-15 secondes (acceptable)
- Latence acces ulterieur : <50ms (lecture fichier)

Livrables :

- `src/shinobi/i18n/wiki_translator.py`
- Tests : `tests/unit/test_wiki_translator.py` (avec mock Qwen)

Critere de sortie :

- Top-100 chars : `data/i18n/wiki/<lang>/<id>.json` existe pour les 7 langs cibles
- Fallback Qwen : la 1ere lecture pour un char hors top-100 produit un cache valide
- Hors-ligne : si Qwen down, retourne FR original avec marqueur `_translation_pending: true`

### Phase i18n.8 : Player input strategy (0.5 jour)

Objectif : convertir le texte joueur (objectifs, intentions, dialogues custom) vers la langue config.

Methode :

1. Au moment de la saisie joueur (api ou CLI) :
   - Detection langue via Qwen3-4B (call ~50 tokens, ~200ms)
   - Si langue detectee == langue config : stockage verbatim
   - Sinon : Qwen translate vers langue config + cache dans le payload du goal/intent

2. Dans `Goal` schema :
   ```python
   description_player: str  # verbatim joueur
   description_player_original_language: str  # detectee (ex: "fr")
   description_player_translated: dict[str, str]  # {"en": "...", "ja": "..."}
   interpretation_canonical: str  # toujours dans langue config
   ```

3. Affichage : si lang config existe dans `description_player_translated`, l'utiliser; sinon afficher original avec marqueur

Livrables :

- `src/shinobi/i18n/player_translator.py`
- Modification `src/shinobi/goals/declaration.py` : `Goal` schema enrichi
- Modification `src/shinobi/cli/play.py` `/declare` flow : auto-detect + translate
- Modification API `POST /play/{id}/goals` : auto-detect + translate
- Migration script `scripts/migrate_goals_i18n.py` pour les saves existantes
- Tests : `tests/unit/test_player_translator.py` (8 tests)

Critere de sortie :

- Goal cree en FR avec lang=en : `description_player_translated["en"]` rempli
- Affichage du goal en mode EN montre la version EN
- Pas de regression sur les 1524 tests
- Goals existants dans saves sont migres au prochain `/play/{id}/initialize`

### Phase i18n.9 : API Accept-Language + i18n responses (0.5 jour)

Objectif : l'API respecte le header HTTP `Accept-Language`.

Methode :

1. Middleware FastAPI qui parse `Accept-Language: en-US, en;q=0.9, fr;q=0.8`
2. Si header present : `set_language(<langue prioritaire supportee>)` pour la duree de la requete
3. Sinon : utilise `preferences.json` global
4. Toutes les routes existantes utilisent `t(...)` pour leurs strings

Modifications :

- Schemas Pydantic : `description_fr` -> `description` (resolu au runtime selon lang)
- Routes canon : `name_romaji` reste tel quel; `name`/`description` renvoyes en langue active
- Wiki sections : routes `/canon/characters/{id}/wiki` chargent `data/i18n/wiki/<lang>/<id>.json` ou fallback
- OpenAPI doc : `summary` et `description` des routes traduites; FastAPI ne supporte pas nativement la doc multilingue, donc on utilise la langue par defaut du serveur (EN) pour `/docs`. Si client veut FR, la doc OpenAPI reste EN mais les responses sont FR.

Livrables :

- `src/shinobi/api/middleware/i18n.py`
- Modifications schemas dans `src/shinobi/api/schemas.py`
- Modifications routes dans `src/shinobi/api/routes/*.py`
- Tests : `tests/unit/test_api_i18n.py` (12 tests, un par langue + middleware + fallback)

Critere de sortie :

- `GET /canon/characters/uchiha_itachi` avec `Accept-Language: ja` retourne `name` en romaji + `description` traduite JA
- Sans header : utilise `preferences.json` global
- 132 tests API existants pass (avec lang=en par defaut)

### Phase i18n.11 : Tests cross-langue (1 jour)
